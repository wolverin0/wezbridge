Loop Engineering Playbook

When to let agents run themselves on a goal — and when not. A reusable decision method + the streamlined loop, with a worked example on Futura CRM.
2026-06-25 · grounded in the loop-engineering research corpus (51 sources) + direct inspection of the crm/ repo.

"Loop engineering" = you stop hand-prompting an agent turn-by-turn and instead design a goal-driven, self-correcting cycle that prompts the agent for you and runs until an independent check says it's done. You become the architect of the loop, not the operator inside it.

But the method is only worth using on work that fits it. Most of the value — and all of the danger — is in one question.

1 · The one gate that decides everything

Can a machine decide "done" without a human looking?

If yes → the work is loopable. If no → looping it just means an agent grading its own homework, which is the #1 fatal failure mode (the "self-approval trap"). A loop with a weak stop condition doesn't save you work — it produces expensive, confident garbage faster.

Everything else is secondary to having a real stop condition: a deterministic, independent signal that the goal is met. Tests passing, a build succeeding, a linter clean, a coverage number crossed, an eval grader scoring ≥ N, a schema validating, an HTTP 200. The agent that does the work must never be the thing that decides it succeeded.

The 5-axis suitability rubric

Score a piece of work before looping it. Axis 1 is a hard gate — fail it and the rest don't matter.

AxisLoop it (green)Be careful (amber)Don't loop (red)
1. Stop condition hard gate strong Deterministic + independent: tests, build, lint, coverage %, eval grader, schema check weak Partial: a few tests but big gaps; proxy metric only none "Looks good", taste, UX feel, "is this the right feature?"
2. Blast radius reversible git-tracked code, sandbox, branch, throwaway DB contained staging data, internal tools irreversible prod data, payments, sending messages to customers, schema drops
3. Decomposability scoped Splits into small tasks each independently verifiable coupled Stages depend on each other but are sequenceable monolithic One big judgment call that can't be chunked
4. Spec clarity written A spec/roadmap/audit list exists; "done" is describable sketch Rough intent, needs a planning pass first vague "Make it better" — no one can say what good is
5. Cost ceiling capped You can set MAX_ITER + token/$ budget and walk away watch Expensive per iter; needs a tight cap unbounded Each iter hits a paid external API with no cap

2 · The decision tree

Work arrives →
  Q1. Is there a command that returns pass/fail without a human?
    ├─ No  → don't loop keep a human in the loop, OR first build the check (write tests/an eval), then revisit.
    └─ Yes → Q2. Can a bad iteration be thrown away safely (git/sandbox/branch)?
        ├─ No  → don't loop the action is irreversible (prod data, payments, customer messages).
        └─ Yes → Q3. Did a single manual run already succeed end-to-end?
            ├─ No  → not yet do 1 manual run + 3–5 manual repeats first. Never automate an unproven run.
            └─ Yes → loop it wire Goal→Execute→Verify→Iterate with MAX_ITER + budget cap, run unattended, review by PR.

Q3 is the most-skipped step and the most expensive to skip. If you can't get it working once by hand, a loop just reproduces the failure N times and bills you for it.

3 · The streamlined method

The cycle

GOAL      one measurable objective
  │
EXECUTE   maker agent, fresh context,
  │       state loaded from disk
VERIFY    independent check runs
  │       (maker ≠ checker)
  ├─ pass → commit + update memory → STOP
  └─ fail → record why → ITERATE
            (until MAX_ITER / budget)

The 5 rules that make it work

  • Independent verification — the checker is a different process/prompt than the maker. Deterministic where possible.
  • Stateless iterations — each turn starts fresh; state lives in files/git, not chat history. Beats context rot.
  • Scoped workers — narrow task + narrow tools > one giant agent with everything.
  • Proof + memory — every pass emits an artifact (test log, diff) and writes what it learned.
  • Hard rails — MAX_ITER, token/$ cap, and a human gate on anything irreversible.

The reusable per-project loop spec

Drop one of these in <project>/.loops/<name>.md. It's the whole contract — fill the blanks and it's runnable.

# Loop: <name>

Goal:            <one measurable sentence — what "done" means>
Trigger:         <manual | on CI red | cron | when audit-report changes>
Maker:           <model that does the work>
Checker:         <the INDEPENDENT verification — a command, not an opinion>
Stop condition:  <exact deterministic gate, e.g. "pytest exits 0 AND coverage ≥ 80">
Max iterations:  <N>
Budget cap:      <tokens / $>
State location:  <dir/files the loop reads at the start of each turn>
Memory update:   <where results + lessons are written>
Human checkpoint: <none | review PR at end | approve before any external action>
Blast radius:    <branch/sandbox — what a bad iteration can and cannot touch>

Minimal stateless runner (shape, not framework)

while iter < MAX_ITER and spent < BUDGET:
    state   = load_relevant_files()        # only what's needed, fresh
    result  = maker.run(goal, state)       # new context every turn
    proof   = checker.run()                # INDEPENDENT, deterministic
    if proof.passed:
        git.commit(result); memory.write(proof); break
    else:
        memory.append_failure(proof.why)   # next turn reads this
        iter += 1
# exit → human reviews the branch / PR

You don't need a heavyweight tool to start. The "Ralph" family (ralph-orchestrator, ralphy, ralph-zero) and skills like loop-me / the loop-library are just polished versions of this while loop. Start with the loop; adopt a harness only when you feel the pain.

4 · Worked example — Futura CRM strong fit

Why this project is a clean teaching case: it already carries the verification surface most projects lack. Inspected in crm/recovered-source/crm-standalone/:

Verification surface (already in repo)What it gives you as a stop condition
pytest · 117 test files · asyncio · integration markerDeterministic backend gate — "all tests green" is machine-checkable.
CI: ci.yml, api-sdk.yml, sec-scan.ymlA canonical, already-agreed definition of "green" — the loop's checker can just be "make CI pass."
backend/evals/ — dataset.py, grader.py, replay.py, golden/A maker/checker harness that already exists. Golden dataset + grader = independent semantic verification for LLM-ish behavior, not just unit tests.
Frontend: tsc -b && vite build + eslint .Type-check + build + lint = a deterministic frontend gate.
audit-report.md, MONITORING_PLAN.md, RUNBOOK.mdPre-written work-lists → ready-made loop goals with describable "done".

What we would not loop on Futura CRM

  • no Sending WhatsApp / outreach to real customers — irreversible external action; your own rule already makes this operator-approval-only.
  • no Anything touching leads.db / production data or payments — blast radius fails axis 2.
  • no "Is this the right feature / does this UX feel good?" — no machine stop condition; needs your taste. (A loop can build a feature once you've decided it and written its acceptance test — it can't decide it.)
  • no Schema migrations with irreversible drops — gate behind a human even if reversible-on-paper.

So why would you actually use this here?

Because the CRM accumulates a large surface of verifiable maintenance — coverage gaps, audit findings, eval regressions, flaky CI — that is real work, is boring, and has a built-in "done" signal. That's the sweet spot: you define the target once (e.g. "audit-report has zero P0/P1, CI green"), the loop grinds it overnight against the existing tests/CI/evals, and you review a PR in the morning. You stop being the loop for the parts a machine can already grade — and you keep your attention for the feature and customer decisions a machine can't.

5 · When not to use it (any project) + fatal anti-patterns

Don't loop when…

  • There's no independent "done" check (and you're not willing to build one first).
  • The action is irreversible or customer-facing (send/publish/pay/delete prod).
  • The task is a single design/taste judgment that can't be chunked.
  • You haven't gotten it working once by hand.
  • You can't put a hard cap on iterations and spend.

Fatal anti-patterns

  • Self-approval trap — maker grades its own work.
  • Doom loop — one fat context, logs pile up, context rot, agent spins.
  • Monolith agent — every tool, no scope, drifts off-goal.
  • Vague goal — "make it better" has no stop condition.
  • No caps — runs until the bill, not the goal.
  • Looping the unproven — automating a run that never worked manually.

How this ties back to wezbridge / omniclaude: the orchestrator pattern people use (Derek Nee's "Matrix": Workspace Brain → Runtime → Department leads → scoped Claude Code / Codex worker seats → Proof / Check-in loop) is exactly the wezbridge pane model with a stop condition bolted on. A pane is a worker seat; the orchestrator pane dispatches a loop spec; the project's tests/CI/evals are the checker. The missing piece to operationalize is small: a per-project .loops/ spec + a stateless runner that reads state, runs the maker in a worker pane, runs the checker command, and advances only on proof.
