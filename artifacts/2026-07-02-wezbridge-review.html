wezbridge — project review

v3.4.2 · zero-dep MCP server + :4200 daemon · reviewed 2026-07-02 · evidence: graphify report, 263/263 tests, live daemon probe, source read of mcp-server / handlers / routes
Verdict: the core is healthy — 263/263 tests green, clean module split after the v3.4 refactor (handlers/, routes, ipc), the audit-remediation wave (b242ea2) clearly paid off. But the repo has drifted from its own documentation, the daemon was found dead with nothing supervising it, one LAN-exposure security hole survived the audits, and the two worst real-world failure modes (enter-key loss, event-loop blocking) are documented as protocol rules instead of fixed at source.

🛑 Broken / fix first

CRITICAL :4200 binds all interfaces with auth disabled in dev

src/dashboard-server-routes.cjs:286 — server.listen(PORT) has no host arg → binds 0.0.0.0. checkAuth() (line 34-36) returns allow-all when WEZBRIDGE_API_TOKEN is unset and NODE_ENV !== 'production' — which is every real run. The Origin-header CSRF defense only stops browsers; any device on your LAN can POST /api/sessions/:id/prompt and type arbitrary text + Enter into your panes — including shell panes = remote command execution. The safety-policy 5-rule gate filters known-dangerous patterns, not everything.

Fix: server.listen(PORT, '127.0.0.1') by default; add WEZBRIDGE_BIND env for the rare LAN case and REQUIRE the token whenever bind ≠ loopback. ~5 lines + 1 test.

HIGH Daemon is down and nothing notices

Live probe: curl :4200/api/panes → connection refused. Core MCP tools survive (they call wezterm CLI directly now), but everything the daemon owns dies silently: telegram-streamer, SSE, auto-handoff, grades, and session-snapshot (default ON since v3.4.1 — you currently have no crash-restore snapshots). install-autostart.cmd and start-wezbridge-sentinel.cmd exist but evidently aren't active.

Fix: (a) run install-autostart so the sentinel supervises the daemon; (b) add a bridge_health MCP tool (wezterm reachable? daemon up? version) so any session can self-diagnose in one call; (c) auto_handoff should return "daemon down — run npm run dashboard" instead of a raw fetch error.

HIGH CLAUDE.md is lying to every session that loads it

CLAUDE.md says v3.3.0 / 184 tests / "the dashboard daemon must be alive for the MCP server's tools to work" / a repo layout without handlers/, session-snapshot, project-status-registry, telegram-router, or the dashboard-server split. Reality: v3.4.2, 263 tests, MCP talks to wezterm directly (only auto_handoff touches :4200). Every new session inherits these wrong facts — this review almost started from them.

Fix: rewrite CLAUDE.md's status header, layout table, and the daemon-dependency sentence. 15 min, highest doc ROI in the repo.

HIGH Untracked TypeScript orphans in src/

src/backend/{peers-broker,peers-channel-plugin,wezterm-bridge}.ts, src/frontend/{XtermPane.tsx,scrollback.ts}, src/mcp/handlers/peers.ts — TS/React files in a zero-dependency CJS project, imported by nothing, and not committed (only src/shared/types.ts is tracked). One git-clean and they're gone forever; meanwhile graphify indexes PeersBroker as a god node, polluting the architecture picture.

Fix: decide once — commit to an experiment/peers-broker branch (or move under docs/_drafts/) if it's the future xterm.js direction, else delete. Don't leave unrecoverable work sitting untracked.

⚠ Needs tweaking

MED Enter-key loss is patched by protocol, not by code (the claim-8945 bug)

send_prompt does sendText(text) + sendTextNoEnter('\r') and hopes. The known failure (Enter not registering) is "fixed" by a hard rule in three CLAUDE.mds forcing every caller to follow with send_key('enter') — and the A2A dispatch prompt in event-handlers.cjs even teaches peers the two-call dance.

Fix at source: after sending, wait ~800ms, read the pane tail; if the prompt text is still sitting unsubmitted in the input box, resend \r (max 2 retries) and return {submitted: true|false}. Then delete the protocol rule everywhere. This kills the #1 A2A failure mode.

MED spawn_session / wait_for_idle block the whole MCP server

Both use execFileSync sleep loops (spawn: 2s shell-init + up to 8s boot-wait; wait_for_idle admits it in a comment at mcp-server.cjs:663). The stdio JSON-RPC loop is single-threaded — while one tool sleeps, every other tool call from the session stalls behind it. An orchestrator spawning 3 peers serializes ~30s of dead time.

Fix: make the handlers async (await setTimeout) — JSON-RPC responses can resolve out of order, nothing else needs to change.

MED spawn_session is missing the params you actually need

  • No model param — your own global CLAUDE.md carries a workaround note ("spawn_session has no model param — note in prompt if the executor should self-restrict"). Executor panes should be claude --model sonnet per your tiering rule.
  • No agent param — can't spawn a Codex pane via MCP even though the repo bills itself as the Claude/Codex bridge; you shell out manually today.
  • --continue is the default — resumes whatever session last ran in that cwd; a "new peer" can wake up inside an old conversation. Persona spawns already avoid this deliberately (the comment explains why). Make fresh-start the default, add continue: true opt-in.
